# C2C Plat - Developer Guide

## Introduction
Welcome to the GloMart (formerly C2C Plat), a modern Customer-to-Customer (C2C) e-commerce marketplace designed to effortlessly connect buyers and sellers. This guide explains how the backend PHP system integrates seamlessly with the dynamic frontend interface.

---

## 1. System Architecture

The project has been refactored from a static HTML prototype into a fully functional, database-driven PHP web application.

- **Frontend layer:** HTML5, CSS3, Bootstrap 3.4.1, jQuery, and OwlCarousel2 for dynamic sliders.
- **Backend layer:** Pure PHP 8 (procedural logic interacting with object-oriented database models).
- **Database layer:** MySQL (`c2c_ecommerce` database) connected via `mysqli` in the `DBController`.
- **AJAX layer:** Asynchronous Javascript (`app.js`) talks to `cart_action.php` for seamless "Add to Cart" and quantity update experiences without page reloads.

---

## 2. Directory & File Structure

### Root PHP Files (Pages)
- `index.php`: The homepage. Displays hero banner, category overview, and two dynamic product carousels ("Brand New" and "Sale").
- `header.php` & `footer.php`: Global templates included on every page. The header contains the top navigation bar, dynamic cart badge, and search form.
- `login.php`, `register.php`, `logout.php`: User authentication and registration system with secure session handling.
- `phones.php`, `pc_parts.php`, `peripherals.php`, `accessories.php`, `shop.php`: Category-specific and catalog pages that load their respective products in a grid.
- `product_detail.php`: Dynamically renders product specs, images, pricing, stock status, seller info, and related products.
- `cart.php` & `checkout.php`: The shopping cart and digital wallet checkout view.
- `wallet.php`: Digital wallet dashboard for users to add mock funds.
- `track_orders.php`: Customer-facing dashboard displaying past purchases and order status.
- `my_listings.php`, `add_listing.php`: Seller listing dashboard (CSS Grid layout) and KYC listing form — restricted to users with `role = seller`.
- `submit_listing.php`: Backend AJAX handler for processing and saving new listings.
- `become_seller.php`: Allows buyers to apply for a seller account. Sets user `status` to `pending`. Admin approves/rejects from the dashboard.
- `admin.php`: The admin dashboard. Shows active users, pending seller applications, seller listings moderation queue, and user metrics.
- `admin_action.php`: Admin AJAX endpoint for metrics, approving/rejecting listings, and approving/rejecting seller applications.

### Logic & Action Handlers
- `functions.php`: Included globally. Starts the PHP `session_start()`, creates database model instances (`$product`, `$cart`, `$user_obj`), and provides rendering helper functions (`renderProductCard()`, `renderCartRow()`).
- `cart_action.php`: A dedicated JSON API endpoint handling all AJAX requests from `app.js` (actions: `add`, `update`, `remove`, `clear`, `count`).

### Database Classes (`/database/`)
- `c2c.sql`: Complete MySQL dump to build tables (`product`, `user`, `cart`, `wishlist`, `transaction`, `pending_listing`) and seed initial data.
- `DBController.php`: Manages the MySQL connection.
- `product.php`: Handles fetching products, filtering by categories, flags (sale/new), and performing searches.
- `Cart.php`: Handles cart interactions and checkout processing (digital wallet integration).
- `User.php`: Handles user management, authentication, wallet balance, and transaction history.
- `Listing.php`: Handles inserting and retrieving pending KYC seller listings.

### Assets & Styling
- `app.js`: Main Javascript file managing Owl Carousels, AJAX add-to-cart clicks, cart quantity buttons, and client-side category filtering.
- `admin.js`: Manages the client-side search bar and status dropdowns inside the `admin.php` page.
- `style.css` & `admin.css`: Core stylesheets defining the brand identity and custom UI components.
- `/assets/`: Directory containing all product, banner, and UI images.

## 3. How the Data Flow Works

### Rendering Products
Unlike the prototype that used hardcoded JSON arrays, products are now fully driven by the PHP server.
1. A page (e.g., `phones.php`) calls `$product->getProductsByCategory('phones')`.
2. The page then loops through the resulting array and passes each item to `renderProductCard($item)` defined in `functions.php`.
3. `renderProductCard()` echoes pure HTML containing the product image, title, price, and the exact `item_id` attached to the "Add to Cart" button as a data attribute (`data-item-id`).

### Add to Cart (AJAX)
1. User clicks the "Add to Cart" button.
2. `app.js` intercepts the click, extracts the `item_id`, and fires an AJAX POST request to `cart_action.php?action=add`.
3. `cart_action.php` calls `$cart->addToCart($user_id, $item_id)`. If the item exists in the cart, the quantity increments; otherwise, a new row is inserted.
4. The endpoint returns JSON containing the new total cart count.
5. `app.js` updates the cart icon badge in the header without refreshing the page.

### Checkout & Digital Wallet Flow
1. User navigates to `checkout.php` to review their order.
2. The page verifies the user's `wallet_balance` via the `User` class.
3. Upon confirming payment, `$cart->checkoutCart()` is called. This initiates a MySQL transaction.
4. The buyer's balance is deducted, the seller's balance is increased.
5. A new row is inserted into the `transaction` table, the product `item_status` is marked as `sold`, and the cart is cleared.

### Seller KYC Verification Flow
1. Seller navigates to `add_listing.php`, fills in item details, uploads an image, and inputs their ID Number.
2. On submit, JavaScript calls the external `VerifyNow` API endpoint to verify their identity.
3. If successful, the form data along with the uploaded image (via `FormData`) and the KYC JSON data is POSTed to `submit_listing.php`.
4. `submit_listing.php` securely moves the uploaded image file into `/assets/uploads/` and uses the `Listing` class to insert the payload into the `pending_listing` table for admin review.

### Admin Dashboard & Moderation
1. The Admin Dashboard (`admin.php`) utilizes `$user_obj->getActiveUsers(15)` to display a real-time list of users who have been active within the last 15 minutes.
2. **User Metrics:** When an admin clicks on any user row, a JavaScript event listener triggers an AJAX request to `admin_action.php?action=get_metrics`. This fetches their Total Bought, Total Sold, and Preferred Category natively calculated via `JOIN` clauses on the `transaction` and `product` tables, displaying the data in a clean Bootstrap modal.
3. **Seller Moderation Queue:** Below the active users table, the dashboard displays listings pending approval (fetched via `$listing->getAllPendingListings()`). 
4. Admins can view the item details and the seller's KYC Match Score. Clicking "Approve" triggers an AJAX request to `admin_action.php?action=approve_listing`, which securely initiates a MySQL transaction, transfers the pending item into the live `product` table, and marks the pending record as approved. Clicking "Reject" simply updates the record status to rejected.

## 4. How to Extend the Platform

### Adding a New Product
To add a new product, you simply insert a new row into the `product` table in your MySQL database via phpMyAdmin or the MySQL command line.
The site will immediately and automatically display it on the correct category page, and if `item_is_new` or `item_is_sale` is set to `1`, it will appear on the homepage carousels.

### Adding a New Category Page
1. Create a new PHP file (e.g., `tablets.php`).
2. Include `header.php` and `footer.php`.
3. Call `$product->getProductsByCategory('tablets')`.
4. Render the grid using `renderProductCard()`.
5. Update `header.php` to include a link to `tablets.php` in the navigation menu.

## 5. Security & Best Practices Implemented
- **Prepared Statements:** The `Cart` and `User` database classes utilize `bind_param` heavily to prevent SQL injection vulnerabilities.
- **Passwords:** The `User` class is strictly configured to use modern, native PHP hashing functions (`password_hash` with `PASSWORD_DEFAULT` and `password_verify`). MD5 and SHA1 have been completely avoided.
- **XSS Prevention:** Outputs (like product titles and user names) inside `renderProductCard()` and `admin.php` are wrapped in `htmlspecialchars()` to prevent Cross-Site Scripting.
